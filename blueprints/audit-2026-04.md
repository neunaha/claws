# Claws Audit — April 2026

Deep review of the codebase from a non-technical-user POV plus an engineering lens. Covers concrete bugs, architectural drift, comparison to similar OSS tools, the memory/knowledge-sink question, a user-mistake taxonomy, and a prioritized fix list.

Scope of review: `extension/src/extension.js` (492 lines), `mcp_server.js` (398), `scripts/install.sh` + `install.ps1`, `scripts/shell-hook.sh`, `scripts/terminal-wrapper.sh`, `docs/protocol.md`, `extension/package.json`, the v0.4 rewrite plan, every `/claws-*` slash command.

---

## A. Bugs and architectural flaws

Concrete, line-referenced. Each labeled by severity: **CRIT** (security/data loss), **HIGH** (broken feature), **MED** (silent degradation), **LOW** (polish).

### A1. Protocol: `id` collides with itself — **HIGH**

`docs/protocol.md` lines 88, 99, 117, 173 show requests like `{ "id": N, "cmd": "send", "id": "5" }` — the same JSON object has two `id` keys (request id + terminal id). JSON parsers keep the last one. In `extension.js:379` the response uses `req.id` — which is now the *terminal* id, not the request id. The reason this hasn't broken anything: `mcp_server.js` opens a fresh socket per call, so requests never multiplex. Any client that tried to pipeline would see wrong-response routing.
**Fix:** rename request id to `rid`; keep `id` for terminal id.

### A2. The "safety gate" is documented but doesn't exist — **HIGH** (marketing/code mismatch)

`README.md:152` and the architecture text in `CLAUDE.md` describe a foreground-process detection that warns before sending into a TUI. Searching `extension.js` — no such code, no `safety` module, no foreground process detection.
**Fix:** implement it (read `/proc/$pid/comm` on Linux, `ps -o comm=` on macOS, etc.) OR delete the marketing claim.

### A3. Shell-hook RCE via argument injection — **CRIT**

`scripts/shell-hook.sh:115-128` (`claws-run`) interpolates `$id` and `$sock` directly into a `node -e "..."` heredoc:

```bash
node -e "
... 
const sockPath='$sock',termId='$id',cmdFile='$tmpf';
..."
```

Concrete attack:
```bash
claws-run "$(printf '\x27); require("child_process").exec("rm -rf ~"); (\x27')" "echo ok"
```
The `'` in the argument closes the JS string and injects arbitrary node code. Same class of bug exists in `claws-ls`, `claws-new`, `claws-log`.
**Fix:** pass arguments via env vars or stdin, not string interpolation.

### A4. Bracketed-paste claim is false — **MED**

`extension.js:235`: `t.sendText(req.text, true)`. VS Code's `sendText` does NOT wrap in bracketed paste. Multi-line prompts fragment line-by-line in many shells. README and tool descriptions both lie about this.
**Fix:** wrap in `\x1b[200~ ... \x1b[201~` manually before `sendText`, OR delete the claim.

### A5. Extension version hardcoded in installer — **HIGH** (time-bomb)

`scripts/install.sh:51`: `EXT_LINK="$EXT_DIR/neunaha.claws-0.1.0"`. `extension/package.json` says `"version": "0.1.0"` — they happen to match today, but on any version bump they desync. VS Code requires the directory name to match `<publisher>.<name>-<version>`. Bump version → install creates wrong directory → VS Code refuses to load.
**Fix:** derive version from `package.json` at install time.

### A6. `node -e` interpolation injection in install.sh — **CRIT**-class but rare

`install.sh:135`: `args: ['$MCP_PATH']`. Same single-quote injection class as A3. If `$HOME` or `$INSTALL_DIR` ever contains a `'`, install corrupts `settings.json`. Realistically rare but real.
**Fix:** write JSON to a temp file, then `node` reads it via `JSON.parse(fs.readFileSync(...))`.

### A7. Workspace socket collision — **HIGH**

Open the same workspace in two VS Code windows → window 2's `fs.unlinkSync(socketPath)` (`extension.js:358`) destroys window 1's socket. Window 1's clients silently fail. No detection, no warning.
**Fix:** before unlinking, try connecting to the existing socket. If it responds, refuse to start with a clear error: "Another VS Code window owns this workspace's socket."

### A8. Output truncation drops the BEGINNING — **HIGH**

`extension.js:142-143`:
```js
if (state.output.length > cap * 2) {
  state.output = state.output.slice(-cap * 2);
}
```
For a 10-minute build that prints errors in minute 1, the errors are GONE by minute 9. AI sees "build failed" with no clue why.
**Fix:** keep both head and tail with a `[...skipped X bytes...]` marker in the middle, OR write full output to a file and return the truncated view + the file path.

### A9. `script(1)` on Windows = silent failure — **HIGH**

`terminal-wrapper.sh` is a bash script. `install.ps1` doesn't ship a Windows equivalent. On Windows, `claws_create wrapped=true` opens a regular shell with no logging — same name "Claws Wrapped" but no actual wrapping. `claws_read_log` returns "not wrapped" with no explanation.
**Fix (interim):** detect Windows in `extension.js` and refuse `wrapped=true` with a clear "wrapped terminals require macOS/Linux until v0.4 ships ConPTY support."

### A10. PowerShell shell hook is decoration only — **MED**

`install.ps1:124-136` adds a banner to the user's PowerShell profile but **no `claws-ls` / `claws-new` / `claws-run` / `claws-log` functions**. Windows users get the logo and nothing usable.
**Fix:** port the four shell functions to PowerShell.

### A11. No retry, no health endpoint — **MED**

`mcp_server.js:80` calls `net.createConnection(sockPath)` with no retry. If the extension is mid-restart, every tool call fails. There's no `ping` or `health` command in the protocol.
**Fix:** retry 3× with 250ms backoff in `mcp_server`. Add `ping` command in extension.

### A12. `claws_worker` hardcodes a 5s boot wait — **MED**

`mcp_server.js:328`: `await sleep(5000)`. On slow machines Claude isn't ready → mission goes to the shell. On fast machines, 4500ms wasted per spawn.
**Fix:** poll the wrapped terminal's log for the Claude TUI's prompt marker (pexpect-style). Replace fixed sleep everywhere with `claws_wait_for(id, pattern, timeout)`.

### A13. `pendingWrappedProfiles` array never expires — **LOW**

`extension.js:437` pushes; cleanup only happens in the `onDidOpenTerminal` adopt path. Click "Claws Wrapped Terminal" then click away → entry stays forever.
**Fix:** time-out entries after 30s; expire on `provideTerminalProfile` retry.

### A14. ANSI strip misses OSC sequences — **LOW**

`extension.js:33`: regex covers CSI but not OSC (`ESC ] ... BEL` / `ESC ] ... ESC \`). Hyperlinks, terminal titles, and modern shell prompts leave junk in stripped logs.
**Fix:** add OSC pattern. Reference: xterm.js's `parser.ts` for canonical patterns.

### A15. Activation event runs on every workspace open — **LOW**

`extension/package.json:41`: `onStartupFinished`. Even users who never call Claws pay the activation cost on every window. ~50-100ms per window.
**Fix:** activate on first command/socket request — `onCommand:claws.status` + `workspaceContains:.claws/` activation events.

### A16. `.claws/` is committed to git by default — **CRIT** (privacy)

`.claws/` lives in workspace. Most users won't think to gitignore it. First commit leaks pty logs (potentially passwords typed in `sudo`, ssh keys, tokens in `git push` errors).
**Fix:** on first activation, write `.claws/.gitignore` containing `*` and `!.gitignore`. Idempotent. Cost: ~5 lines.

### A17. Zero tests — **HIGH** (process)

`extension/test/` doesn't exist. No CI. Every change is a YOLO ship. v0.4 plan mentions adding tests but Phase 1 prerequisites it on the rewrite — so tests don't exist for the current code that users actually run.
**Fix:** smoke-test scaffolding using `@vscode/test-electron` for activation + socket. Ship before any rewrite work.

### A18. Installer aggressively removes ALL `neunaha.claws-*` symlinks — **MED**

`install.sh:95`: `rm -f "$EXT_DIR"/neunaha.claws-*` removes any version, including a manually-installed VSIX. Hostile to power users who pin versions.
**Fix:** only remove the version we're about to install.

### A19. "Claws Wrapped Terminal" identification is by name — **LOW**

`extension.js:409`: `/^Claws Wrapped (\d+)$/.exec(t.name || '')`. Fragile — VS Code may auto-rename on collision; user can rename a terminal manually and break the matcher.
**Fix:** use a unique env var (`CLAWS_TERMINAL_TOKEN=<uuid>`) set in the profile env, read back via terminal env (when API supports it) or via the wrapper script writing the token to a known file.

### A20. Installer warns about prerequisites and continues — **HIGH**

`install.sh:74-79`: "Node.js not found — some features (MCP server) may be limited" then continues. Result: install "succeeds" with all green checks but the MCP server is unusable. False confidence is the worst failure mode.
**Fix:** halt on missing prerequisites. Print platform-specific install commands. (See `zero-prereq-install.md`.)

## C. Memory + Obsidian — recommendation

**Don't tie it to Obsidian. Build a "knowledge sink" with a markdown directory. Obsidian users get a vault for free; everyone else gets greppable notes.**

### Why the Obsidian-specific framing is wrong
- Locks out Logseq / RemNote / Notion / plain-md users.
- Conflicts with Claude Code's existing native auto-memory at `~/.claude-campus/projects/.../memory/` — users will ask "where do I look?"
- Scope creep — Claws is a terminal control bridge, not a notes app.
- Privacy hazard — terminal output ends in markdown files. Easy to accidentally commit secrets.
- Maintenance burden — vault format compat, conflict resolution, Obsidian plugin breakage.

### What to build instead

Two new MCP tools, both opt-in via `CLAWS_KNOWLEDGE_DIR`:

```
claws_remember(category, content, tags?)
  → writes to $CLAWS_KNOWLEDGE_DIR/<category>/<timestamp-slug>.md
  → YAML frontmatter (date, terminal_id, tags), markdown body
  → secret-redaction filter (AWS_*, Bearer , password=, ssh-key patterns)

claws_recall(query, since?, limit?)
  → ripgrep over the directory + score results
  → returns top-N snippets with file paths
```

Plus one ambient behavior, off by default:
- On `claws_close` of a worker terminal, prompt the orchestrator: "anything worth remembering?" If yes → `claws_remember` a one-paragraph summary.

Format is plain markdown with YAML frontmatter — works in Obsidian, Logseq, Foam, plain editors. Obsidian users point a vault at the dir. No Obsidian dep, no marketing dependency, fully optional.

**Scope: v0.5, after the install/reliability work and after the node-pty rewrite. Useful and differentiating, but not core.**

## D. User-mistake taxonomy (52 patterns, 12 categories)

Going wide on the failure modes. Patterns first, then the catalog.

**Cross-cutting patterns:**
1. **Seams.** Most failures happen at boundaries — editor/AI/shell/workspace/network. Wherever two systems hand off, users mismatch them.
2. **Silent fallback.** Whenever code "continues with degraded behavior", the user can't tell what they lost.
3. **Unrecoverable async.** Long-running commands, parallel workers, mid-restart races have no kill/retry/resume primitives.
4. **Implicit state.** Workspace path, env vars, which terminal is "current", which Claude is in PATH — all matter, none are surfaced.

### D1 — Pre-install (1-6)
1. Install on a machine without VS Code → installer "succeeds", extension never loads.
2. Install on Cursor/Windsurf without `CLAWS_EDITOR` → auto-detects, but later editor switch breaks it.
3. Install in a Codespace / devcontainer → socket in container, MCP server on host.
4. `curl | bash` with no integrity check.
5. Install on machine without Node.js → installer warns and continues; MCP unusable. (A20)
6. Two installer runs in parallel → race on git clone, symlink, settings.json.

### D2 — Activation (7-10)
7. Reload VS Code but skip Claude restart → tools missing.
8. Restart Claude but skip VS Code reload → tools loaded but no socket.
9. Reload into a different workspace → socket lives elsewhere.
10. Open new terminal without reload → shell hook works, bridge doesn't.

### D3 — Workspace (11-16)
11. Open Claude Code at `~/` (not a workspace) → `[claws] no workspace folder; bridge disabled` — silently. All tools fail.
12. Two windows on the same workspace (A7).
13. Path with spaces → some shell-hook commands break.
14. User adds `.claws/` to `.gitignore` → fine.
15. User commits `.claws/` (A16) → leaks pty logs.
16. Run Claude as `sudo` → socket created with root uid → user-mode CLI can't connect.

### D4 — Wrapped/unwrapped (17-20)
17. User creates a regular terminal but AI calls `read_log` → fails confusingly.
18. User opens "Claws Wrapped Terminal" but wrapper script is missing → silent fallback to plain shell.
19. User runs `vim`/`htop` in a wrapped terminal → script(1) corrupts Ink/ncurses rendering.
20. User opens Claude Code inside a Claws Wrapped terminal → recursive AI orchestration; undefined behavior.

### D5 — Parallel execution (21-24)
21. AI spawns 5 workers, no naming → all "claws", confusing in panel.
22. Workers never closed → terminal panel fills up; no auto-cleanup.
23. ID collision in protocol (A1) — currently latent; will bite a future client.
24. Two AI sessions both try to `claws_close` same terminal → second fails opaquely.

### D6 — Long-running commands (25-28)
25. AI runs `npm start` via `claws_exec` → blocks; exec times out at 180s; server keeps running silently.
26. Infinite loop → same; no kill primitive from MCP.
27. Output exceeds buffer (A8) → silently drops the beginning.
28. Binary output → ANSI strip leaves UTF-8 garbage.

### D7 — TUI (29-31)
29. AI sends a command into a terminal currently running Claude TUI → text becomes prompt input.
30. AI sends `\r` "to submit a prompt" → if a different dialog is showing, accepts the dialog.
31. AI tries `claws_read_log` on unwrapped terminal → fails; AI doesn't know to recreate as wrapped.

### D8 — Socket / process (32-35)
32. Force-quit VS Code → stale socket; next launch unlinks it but a connect during the gap fails opaquely.
33. `CLAWS_SOCKET` env mismatch → mcp_server can't find socket; "ENOENT".
34. Multiple `claude` binaries in PATH (system + brew + nvm) → installer registers MCP for whichever wins; user runs a different one → no MCP.
35. Symlink in `~/.vscode/extensions/` points at deleted dir → VS Code shows "extension failed to load" with no Claws hint.

### D9 — Update / upgrade (36-39)
36. `/claws-update` while a worker runs → extension restart kills the worker mid-task.
37. `git pull` in `~/.claws-src` without re-running `install.sh` → settings.json points at stale path or missing schema.
38. Major version bump in extension changes protocol → old clients silently break (no protocol version handshake — A11).
39. User has manual edits in `~/.claude/settings.json` → installer's node -e cfg merging clobbers unrelated keys.

### D10 — Auth / security (40-44)
40. `CLAWS_SOCKET` set to a world-readable path → other system users can drive your terminals.
41. Future WebSocket transport launched without TLS on coffee-shop wifi → MITM hijacks terminals.
42. Pty log contains a typed password (sudo/ssh) → log committed to git or synced via cloud.
43. Pty log captures `git push` output containing a token in error messages.
44. AI worker runs `rm -rf` or `git push --force` because the user hallucinated an authorization scope.

### D11 — Mental model (45-49)
45. User confuses Claws (extension) with Claude (the AI).
46. User types `/claws` in a non-Claude terminal — slash commands are Claude-Code-only.
47. User reads "8 MCP tools" as 8 commands and never finds shell-side `claws-ls` etc.
48. User assumes `bash <(curl)` IS Claws — doesn't realize the VS Code extension is required.
49. User uses Claude Code in a terminal *outside* VS Code → no extension → no socket → no tools.

### D12 — Cross-machine (50-52)
50. WSL2: VS Code on Windows host, terminals in WSL → socket in WSL filesystem, MCP server on Windows side → namespace split.
51. Devcontainer: Claws installed inside container BUT extension symlinked on host → mismatch.
52. Remote-SSH: extensions install on remote, socket on remote, but Claude Code runs locally → MCP can't reach.
