# Claws v0.4 — Enterprise Rewrite Plan

## Why this rewrite

v0.1-v0.3 proved the concept: terminal control via a socket bridge works. But the implementation has fundamental architectural problems that patches can't fix:

1. **script(1) wrapping causes TUI glitching** — double-pty layer corrupts Ink/ncurses renderers. Claude Code, vim, htop all show visual artifacts in wrapped terminals. This is the #1 user complaint.

2. **MCP tools don't load until session restart** — the bridge between Claude Code and the extension has no auto-reconnect. Users install, tools are registered, but nothing works until they exit + reopen claude.

3. **Worker lifecycle is incomplete** — `claws_worker` creates + launches + sends but then returns. No auto-monitoring, no auto-cleanup, no completion detection. The orchestrator has to manually poll.

4. **Extension state is in-memory only** — wrapped terminal mappings lost on VS Code reload. Terminal exists, log file exists, but extension forgets the mapping.

5. **No Windows support** — `script(1)` doesn't exist on Windows. ConPTY is the correct approach but requires a custom Pseudoterminal provider.

## The core architectural change

**Replace `script(1)` with VS Code's native Pseudoterminal API.**

Current architecture:
```
VS Code Terminal Panel
└── script(1) process        ← THIS CAUSES THE GLITCHING
    └── /bin/zsh
        └── user's process
```

New architecture:
```
VS Code Terminal Panel
└── ClawsPty (vscode.Pseudoterminal)   ← EXTENSION-NATIVE
    └── node-pty (or child_process)
        └── /bin/zsh
            └── user's process
```

With a custom Pseudoterminal:
- The extension owns the pty directly — no double-layer
- Every byte flows through the extension's `onDidWrite` event — full capture without `script(1)`
- Zero rendering artifacts — what the pty emits is exactly what VS Code renders
- Works on Windows (node-pty supports ConPTY)
- No wrapper script needed
- readLog becomes real-time event streaming, not file-tailing

## The rewrite scope

### Phase 1 — TypeScript + Pseudoterminal (core)

**extension/src/ rewrite:**

```
extension/src/
├── extension.ts          — activation, lifecycle, command registration
├── server.ts             — Unix socket + future WebSocket server
├── terminal-manager.ts   — create/list/close/show with Pseudoterminal
├── claws-pty.ts          — Pseudoterminal implementation (THE KEY FILE)
│                           - spawns shell via node-pty or child_process
│                           - captures all output in a ring buffer
│                           - fires onDidWrite for VS Code rendering
│                           - fires custom events for readLog
│                           - handles resize (SIGWINCH)
│                           - handles close/dispose
├── capture-store.ts      — per-terminal output ring buffer + ANSI stripping
│                           - replaces file-based readLog
│                           - in-memory buffer with configurable max size
│                           - persists to .claws/state.json on write
├── protocol.ts           — request/response types
├── safety.ts             — foreground process detection
└── ansi-strip.ts         — ANSI cleaning (extracted, reusable)
```

**Key dependency**: `node-pty` — native module that provides pty access on macOS/Linux/Windows. Used by VS Code's own terminal, Hyper, and xterm.js. It's the standard.

BUT: node-pty is a native addon requiring node-gyp compilation. This adds a build step. Alternative: use `child_process.spawn` with `{ stdio: 'pipe' }` and a manual pty via `openpty` (more work, zero native deps). For v0.4, use node-pty. For v0.5, evaluate removing it.

### Phase 2 — Full lifecycle worker

**MCP server upgrade:**

`claws_worker` becomes a BLOCKING tool that:
1. Creates a Pseudoterminal-backed terminal (visible, zero glitching)
2. Launches claude --dangerously-skip-permissions
3. Waits for boot (detects welcome banner in capture buffer)
4. Sends the mission prompt
5. Enters a poll loop on the capture buffer:
   - Checks for MISSION_COMPLETE → harvest results → close → return
   - Checks for errors → return error with context
   - Timeout after configurable max (default 30 min)
6. Auto-closes the terminal
7. Returns the full results to the caller

One tool call = complete worker lifecycle. No manual polling. No manual cleanup.

### Phase 3 — State persistence + health check

**State persistence:**
- `capture-store.ts` writes `{terminalId: {logPath, name, created, lastActivity}}` to `.claws/state.json`
- On extension activation, reads state.json and re-associates terminals
- Survives VS Code reload

**Health check:**
- New `claws_ping` MCP tool that returns `{ok: true, version, terminals, uptime}`
- MCP server retries socket connection 3x with 2s delay before failing
- If extension socket doesn't exist, returns clear error: "reload VS Code"

### Phase 4 — esbuild + marketplace publish

**Bundling:**
- esbuild compiles TypeScript + bundles into single `dist/extension.js`
- node-pty bundled as optional dependency (falls back to child_process on Windows without build tools)
- `.vscodeignore` to keep VSIX small

**Marketplace:**
- Create PAT from Azure DevOps
- `vsce publish` from CI
- Users install: `ext install neunaha.claws`
- No more git clone + symlink

### Phase 5 — WebSocket transport

**After marketplace is live:**
- `server.ts` adds WebSocket alongside Unix socket
- Token auth (token shown in Output panel)
- TLS (self-signed auto-generated)
- Same JSON protocol, different transport

## Risk assessment

| Risk | Mitigation |
|---|---|
| node-pty build fails on some systems | Fall back to child_process.spawn with manual pty |
| Pseudoterminal API differences across VS Code versions | Pin to vscode engine ^1.93.0, test on 1.93 + latest |
| Breaking change for existing users | Keep socket protocol identical. Only internal implementation changes. |
| node-pty on Windows without build tools | Bundle prebuilt binaries OR use ConPTY directly via ffi |

## Success criteria

1. Zero visual glitching in any terminal (wrapped or not)
2. `claws_worker` runs full lifecycle with one tool call
3. State survives VS Code reload
4. Install via `ext install neunaha.claws` (marketplace)
5. Works on macOS + Linux + Windows
6. Extension tests pass
